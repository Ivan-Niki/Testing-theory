
==================== ЧТО ТАКОЕ ТЕСТИРОВАНИЕ ====================
Тестирование программного обеспечения (Software Testing) - проверка соответствия между реальным 
и ожидаемым поведением программы, осуществляемая на конечном наборе тестов, выбранном определенным образом.

Тестирование программного обеспечения – это процесс анализа программного средства и сопутствующей документации 
с целью выявления дефектов и повышения качества продукта. (С.Куликов «Тестирование ПО. Базовый курс»).

В более широком смысле, тестирование - это одна из техник контроля качества, включающая в себя активности по
планированию работ (Test Management), проектированию тестов (Test Design), 
выполнению тестирования (Test Execution) и анализу полученных результатов (Test Analysis) с целью определить, 
что компонент (или система) соответствует описанным требованиям, показать, что они пригодны для заявленных целей 
а также для определения дефектов.

==================== ЧТО ТАКОЕ КАЧЕСТВО ПО ===================
-> Качество ПО — это совокупность свойств и характеристик программного обеспечения, которые определяют его
способность удовлетворять требованиям и ожиданиям пользователя.
Оно может быть измерено на основе различных критериев, таких как функциональность, надежность, производительность, удобство использования, безопасность и т.д.

-> Качество (Quality) – cтепень, с которой компонент, система или процесс соответствует зафиксированным 
требованиям и/или ожиданиям и нуждам пользователя или заказчика.


====================== ЦЕЛИ ТЕСТИРОВАНИЯ =====================
Три ключевые цели тестирования:
1. Повысить вероятность того, что разрабатываемое приложение будет работать правильно при любых обстоятельствах.
2. Повысить вероятность того, что разрабатываемое приложение будет соответствовать всем описанным требованиям.
3. Предоставление актуальной информации о состоянии продукта на момент тестирования (предоставление информации о 
статусе готовности заявленной функциональности системы или приложения).

============= КОГДА НАЧИНАТЬ И ЗАКАНЧИВАТЬ ТЕСТИРОВАНИЕ? =============
Критерии входа - это набор условий или требований, которые должны быть выполнены или достигнуты для создания 
подходящих и благоприятных условий для тестирования. Критерии входа, разработанные и утвержденные после 
тщательного анализа программного обеспечения и бизнес-требований, обеспечивают точность процесса тестирования, 
и пренебрежение ими может повлиять на его качество. 

Некоторые из критериев входа, которые обычно используются для обозначения начала тестирования:
- полный или частично тестируемый код доступен (законченность разработки требуемого функционала).
- требования определены и утверждены.
- наличие достаточных и требуемых тестовых данных.
- тест-кейсы разработаны и готовы.
- тестовая среда (окружение) настроена и все другие необходимые ресурсы, такие как инструменты, программные утилиты, конфигурационные файлы и устройства доступны.

Критерии выхода - важный документ, подготовленный командой контроля качества для соблюдения установленных сроков 
и выделенного бюджета. В этом документе указаны условия и требования, которые должны быть достигнуты или 
выполнены до окончания процесса тестирования программного обеспечения. С помощью критериев выхода команда 
тестировщиков может завершить тестирование без ущерба для качества и эффективности программного обеспечения. 

Обычно рассматриваемыми критериями выхода для завершения процесса тестирования являются:
- Сроки вышли или бюджет исчерпан.
- Все тест-кейсы выполнены.
- Желаемый и достаточный охват (покрытие) требований и функциональных возможностей в рамках теста.
- Все выявленные дефекты исправлены и закрыты.
- Не было пропущено ни одной ошибки с высоким приоритетом, высокой серьезностью (severity) или критической ошибки. (Ни одна серьезная или критическая ошибки не остаются открытыми).
- 90-95% всех тестов сделано.

============= QUALITY ASSURANCE =============
Три уровня работы с качеством программного обеспечения: QA, QC, Testing.

Quality Assurance (QA, обеспечение качества) — это активности, направленные на обеспечение уверенности в том, 
что требования к качеству будут выполнены. Это работа над качеством продукта на всех этапах его разработки — 
от планирования до выхода на рынок и встречи с пользователем. 
Основная задача  обеспечения качества – это выстроить систему, которая будет превентивно работать на качество 
продукта, чтобы при тестировании количество дефектов было минимальным. В зависимости от специфики проекта сюда 
может включаться тестирование документации, ревью кода на соответствие стандартам, внедрение каких-то методик по 
работе с качеством, коммуникационные активности и др.


============= QUALITY CONTROL =============
QC (Quality Control) – включает в себя не только сами проверки как таковые, но также подготовку к ним до 
тестирования и анализ полученных результатов после. Составление отчета о качестве продукта перед релизом (что 
нет критических багов), апдейт чек-листов. Соответствует ли это ПО требованиям заказчика, спецификациям, ТЗ и т.д.


============= TESTING ============= 
Тестирование (Testing) – представляет собой проверку создаваемого программного продукта на соответствие требованиям к этому продукту. 
Основная задача тестирования – это выявить и зафиксировать дефекты. Т.е. это проверка какого-то функционала и 
поиск в нём багов (по уже составленным чек-листам и тест-кейсам).



Подведём итог:
1. QA ≠ QC ≠ Testing
2. QA включается в себя QC, который в свою очередь включает в себя Testing.
3. Testing — нахождение ошибок и несоответствий в требованиях, продукте, документации и тд.
4. QC — анализ метрик, статистик, определение и анализ качества продукта.
5. QA — ввод новых практик, которые уменьшают вероятность появления багов в продукте.
6. Тестированием занимается тестировщик.
7. QC обычно выполняет Test Lead либо ключевой тестировщик.
8. QA обычно ложится на плечи QA Manager.


====== ПРИНЦИПЫ ТЕСТИРОВАНИЯ (Testing Principles) ============= 
Принцип 1. Тестирование показывает наличие дефектов (Testing shows presence of defects).
Принцип 2. Исчерпывающее тестирование невозможно (Exhaustive testing is impossible).
Принцип 3. Раннее тестирование сохраняет время и деньги (Early Testing).
Принцип 4. Скопление (кластеризация) дефектов (Defects clustering).
Принцип 5. Парадокс пестицида (Pesticide paradox).
Принцип 6. Тестирование зависит от контекста (Testing is context depending).
Принцип 7. Заблуждение об отсутствии ошибок (Absence-of-errors fallacy).


============= ЧТО ТАКОЕ ВЕРИФИКАЦИЯ и ВАЛИДАЦИЯ =============
Верификация (verification) – доказанное объективными результатами исследования подтверждение того, что определённые требования были выполнены (ISTQB Glossary).
Верификация - это подтверждение путем представления объективных доказательств выполнения данным рабочим элементом установленных требований.

Верификация – это проверки, выполняемые в процессе разработки ПО для ответа на вопрос: «правильно ли мы разрабатываем продукт?». 
Это в т.ч. включает проверку документации: requirements specification, design documents, database table design, ER diagrams, test cases, traceability matrix и т.д. 
Верификация гарантирует, что ПО разрабатывается в соответствии со стандартами и процессами организации, 
полагаясь на reviews и СТАТИЧЕСКИЕ  методы тестирования (т.е. без запуска ПО, но, например, с unit/integration tests). 
Верификация является превентивным подходом (Preventative approach).

Валидация - доказанное объективными результатами исследования подтверждение того, что требования для ожидаемого конкретного использования приложения были выполнены. (ISTQB Glossary).
Валидация - это процесс оценки конечного продукта, чтобы проверить, соответствует ли он потребностям бизнеса и 
ожиданиям клиентов, т.е. отвечает на вопрос: «правильный ли мы разработали продукт?». 
Валидация является ДИНАМИЧЕСКИМ тестированием, т.е. происходит с помощью выполнения кода и прогона тестов на нём (UAT/CAT, usability, всё что угодно). 
Валидация является реактивным подходом (Reactive approach).


====== ПЕРЕЧИСЛИТЕ ХАРАКТЕРИСТИКИ КАЧЕСТВА ПО и ИХ АТРИБУТЫ ======

Выделяют  6 основных характеристик качества ПО, каждую из которых определяют набором атрибутов, имеющих соответствующие метрики для последующей оценки:

1) Функциональность (Functionality) - определяется способностью ПО решать задачи, которые соответствуют 
зафиксированным и предполагаемым потребностям пользователя, при заданных условиях использования ПО. 
Т.е. эта характеристика отвечает за то, что ПО работает исправно и точно, функционально совместимо, 
соответствует стандартам отрасли и защищено от несанкционированного доступа. 
Атрибуты функциональности:  функциональная исправность, соответствие стандартам, функциональная совместимость, безопасность, точность.

2) Надежность (Reliability) – способность ПО выполнять требуемые задачи в обозначенных условиях на протяжении 
заданного промежутка времени или указанное количество операций. 
Атрибуты данной характеристики – это завершенность и целостность всей системы, способность самостоятельно и 
корректно восстанавливаться после сбоев в работе, отказоустойчивость.
Атрибуты надёжности:  завершённость, восстанавливаемость, устойчивость к отказам.

3) Удобство использования (Usability) – возможность легкого понимания, изучения, использования и привлекательности ПО для пользователя.
Атрибуты удобства использования:  удобство изучения, понятность, удобство и простота использования.

4) Эффективность (Efficiency) – способность ПО обеспечивать требуемый уровень производительности в соответствие с выделенными ресурсами, временем и другими обозначенными условиями.
Атрибуты эффективности:  эффективность по времени, эффективность использования ресурсов.

5) Удобство сопровождения (Maintainability) – легкость, с которой ПО может анализироваться, тестироваться, изменяться для исправления дефектов, 
для реализации новых требований, для облегчения дальнейшего обслуживания и адаптироваться к имеющемуся окружению.
Атрибуты удобства сопровождения:  стабильность, анализируемость, контролепригодность, изменяемость.

6) Портативность (Portability) – характеризует ПО с точки зрения легкости его переноса из одного окружения (software/hardware) в другое.
Атрибуты портативности:  удобство установки, заменяемость, совместимость.


=========== Приемка приложения ============

Приемка приложения — это процесс, который включает в себя проверку и оценку программного обеспечения или 
приложения перед его запуском в эксплуатацию. Основная цель приемки — убедиться в том, что приложение 
соответствует установленным требованиям и стандартам качества.

Приемка приложения обычно включает следующие этапы:
-> 1. Тестирование: Проведение различных видов тестирования, таких как функциональное, производительное и 
безопасность, для выявления возможных ошибок и недоработок.
-> 2. Сравнение с требованиями: Проверка приложения на соответствие техническим требованиям и спецификациям, 
которые были согласованы в процессе разработки.
-> 3. Функциональная приемка: Оценка приложения конечными пользователями или заказчиком для подтверждения его 
соответствия их ожиданиям и потребностям.
-> 4. Документирование результатов: Составление отчета о результатах приемки, включая выявленные проблемы, 
недостатки и рекомендации по их устранению.
-> 5. Подписание акта приемки: Если приложение прошло все этапы проверки и удовлетворяет требованиям, подписывается 
акт приемки, что подтверждает его готовность к эксплуатации.
Приемка приложения — важный этап в жизненном цикле разработки ПО, который помогает минимизировать риски и 
обеспечить успешный запуск приложения.


=========== Что такое аудит приложения? ============

Аудит приложения — это систематическая оценка и анализ приложения с целью выявления его недостатков, проблем в 
безопасности, соответствия стандартам и лучшим практикам, а также качества предоставляемых услуг. Аудит может 
включать различные аспекты, такие как функциональность, производительность, безопасность и соответствие 
требованиям.

Основные цели аудита приложения:
-> 1. Оценка безопасности: Анализ уязвимостей приложения, проверка, насколько защищены данные пользователей и 
какая архитектура используется для управления безопасностью.
-> 2. Соответствие стандартам: Проверка, соответствует ли приложение установленным стандартам и нормативным 
требованиям (например, GDPR, HIPAA и др.).
-> 3. Функциональность и производительность: Оценка того, насколько приложение выполняет заявленные функции и 
как оно работает под нагрузкой.
-> 4. Качество кода: Анализ исходного кода на наличие ошибок, недочетов, избыточного кода или небезопасных 
практик программирования.
-> 5. Пользовательский опыт: Оценка удобства интерфейса и взаимодействия пользователя с приложением.
-> 6. Документация: Проверка наличия и качества документации, которая должна сопровождать приложение.


=========== Чем приемка приложения отличается от аудита? ===========

Приемка приложения и аудит приложения — это два различных процесса, хотя они могут пересекаться в некоторых аспектах. Вот основные отличия между ними:

----- Приемка приложения:

1. Цель: Главная цель приемки — определить, соответствует ли приложение установленным требованиям и стандартам, 
чтобы гарантировать его готовность к эксплуатации. Это обычно делается перед запуском приложения.
2. Субъекты: Приемка часто проводится заказчиком или конечными пользователями, которые должны оценить приложение 
с точки зрения его функциональности и соответствия их ожиданиям.
3. Фокус: Процесс сосредоточен на оценке функциональности, удобства использования и соответствия требованиям, 
которые были согласованы в начале проекта.
4. Результат: По итогам приемки составляется акт приемки (или его аналог), который подтверждает готовность 
приложения к эксплуатации.

----- Аудит приложения:

1. Цель: Аудит направлен на всестороннюю оценку приложения, включая анализ безопасности, производительности, 
качества кода и соответствия стандартам. Это может делать как до, так и после запуска приложения.
2. Субъекты: Аудит обычно проводит внутренний или внешний эксперт или группа экспертов, которые могут быть 
независимыми от разработки приложения.
3. Фокус: Процесс охватывает более широкий спектр аспектов, включая стратегические, технические, юридические и 
управленческие, с акцентом на улучшение и выявление уязвимостей.
4. Результат: По итогам аудита создается отчет с рекомендациями по улучшению, устранению уязвимостей или 
несоответствий и повышению общей эффективности приложения.

Вывод:
Приемка приложения сосредоточена на проверке его готовности к эксплуатации и соответствию требованиям, в то 
время как аудит приложения является более глубоким и широким анализом, который охватывает множество аспектов и 
может быть проведен в любой момент цикла разработки приложения, включая пострелизный период.


============= Этапы проведения аудита приложения ==============

Проведение аудита приложения включает несколько ключевых этапов, которые помогают систематизировать процесс и 
обеспечить его эффективность. Вот основные этапы проведения аудита приложения:

1. Определение цели и объема аудита
- Четкое определение целей аудита (например, проверка безопасности, функциональности, соответствия стандартам).
- Определение объема работы, включая функциональность, компоненты и системы, которые будут проверены.

2. Сбор информации
- Сбор всех доступных документов, связанных с приложением, включая техническую документацию, архитектуру системы 
и предыдущие отчеты о тестировании или аудите.
- Интервью с ключевыми заинтересованными сторонами, такими как разработчики, пользователи и администраторы.

3. Анализ исходного кода
- Проведение статического анализа кода для выявления потенциальных уязвимостей, ошибок и нарушений лучших практик программирования.
- Проверка на наличие соответствия стандартам кодирования и документации.

4. Проверка функциональности
- Оценка работы приложения по его функциональным требованиям.
- Проведение функционального тестирования, чтобы убедиться в том, что приложение работает как задумано.

5. Аудит безопасности
- Анализ архитектуры безопасности приложения, в том числе оценка управления доступом, защиты данных и шифрования.
- Проведение тестирования на проникновение (пентест), чтобы выявить уязвимости и потенциальные угрозы.

6. Проверка производительности
- Оценка производительности приложения под различными нагрузками, включая стресс-тестирование и нагрузочное тестирование.
- Анализ метрик производительности, таких как время отклика, время загрузки и потребление ресурсов.

7. Анализ пользовательского интерфейса
- Оценка UX/UI-дизайна приложения с точки зрения удобства использования и доступности.
- Проведение опросов или интервью с пользователями для сбора отзывов о взаимодействии с приложением.

8. Составление отчета
- Сбор всех результатов и выводов в едином отчете.
- Предоставление рекомендаций по исправлению выявленных проблем и улучшению приложения.

9. Презентация результатов
- Презентация итогов аудита заинтересованным сторонам.
- Обсуждение рекомендаций и плана действий для устранения найденных проблем.

10. Мониторинг и контроль
- Последующий мониторинг реализации рекомендованных действий.
- Периодический пересмотр и оценка приложения для обеспечения его дальнейшей безопасности и эффективности.

Эти этапы помогают систематизировать процесс аудита и обеспечить его полноту и эффективное выявление проблем и 
областей для улучшения приложения.




============ Метрики в тестировании =============
Метрики тестирования — это количественные показатели, характеризующие продвижение процессов тестирования ПО, 
уровень качества этих процессов, и производительность QA-команды.
Метрики тестирования используются для отслеживания усилий по обеспечению качества выпускаемого программного 
кода. С их помощью удаётся в численном выражении получить представлении о достижении заданного уровня качества 
или поставленных целей. Визуальное представление результатов формирует наглядную картину процесса тестирования, 
которая может показать узкие места.

Метрики тестирования программного обеспечения подразделяются на два типа:
1) Метрики процесса (Process metrics)
2) Метрики продукта (Product metrics)


------------ Метрики процесса (Process metrics) ------------
Используются в процессе подготовки и выполнения тестирования. Сюда относятся такие метрики как:

-> Продуктивность подготовки тест-кейсов (Test Case Preparation Productivity): используется для расчета количества подготовленных тест-кейсов и усилий (Effort), затраченных на их подготовку.
Test Case Preparation Productivity = (No of Test Case) / (Effort spent for Test Case Preparation)

-> Охват тестового дизайна (Test Design Coverage): процент покрытия тест-кейсами требований.
Test Design Coverage = ((Total number of requirements mapped to test cases) / (Total number of requirements)*100

-> Продуктивность выполнения тестов (Test Execution Productivity): определяет количество тест-кейсов, которые 
могут быть выполнены в час.
Test Execution Productivity = (No of Test cases executed) / (Effort spent for execution of test cases)

-> Покрытие выполненных тестов (Test Execution Coverage): предназначено для измерения количества выполненных 
тест-кейсов по сравнению с количеством запланированных тест-кейсов.
Test Execution Coverage = (Total no. of test cases executed / Total no. of test cases planned to execute)*100

-> Успешные тест-кейсы (Test Cases Passed): для измерения процента пройденных успешно тест-кейсов.
Test Cases Pass = (Total no. of test cases passed) / (Total no. of test cases executed) * 100

-> Неуспешные тест-кейсы (Test Cases Failed): для измерения процента заваленных тест-кейсов.
Test Cases Failed = (Total no. of test cases failed) / (Total no. of test cases executed) * 100

-> Заблокированные тест-кейсы (Test Cases Blocked): для измерения процента блокированных тест-кейсов.
Test Cases Blocked = (Total no. of test cases blocked) / (Total no. of test cases executed) * 100


-------------- Метрики продукта (Product metrics) --------------

-> Уровень обнаружения ошибок (Error Discovery Rate): для определения эффективности тест-кейсов.
Error Discovery Rate = (Total number of defects found /Total no. of test cases executed)*100

-> Процент выявления дефектов (Defect Detection Percentage (DDP)): Количество дефектов, выявленных в фазе 
тестирования, поделенное на число дефектов, найденных в этой фазе тестирования, а также во всех последующих 
фазах.
